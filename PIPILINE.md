# Документация CI/CD Пайплайна проекта «Очумелые ручки»

Настоящий пайплайн разработан на базе платформы **GitHub Actions** и предназначен для автоматизации процессов непрерывной интеграции (CI) и имитации непрерывной доставки (CD) веб-приложения на фреймворке Laravel.

---

## 1. Триггеры запуска (Triggers)

Пайплайн настроен на автоматический запуск при наступлении следующих событий:
*   **Push** (прямая отправка кода) в долгоживущие ветки: `main`, `master`, `develop`, `dev`, `uat`, `qa`.
*   **Pull Request** (запросы на слияние веток) в те же целевые ветки.

---

## 2. Архитектура пайплайна (Jobs)

Пайплайн разделен на две последовательные работы (Jobs): **`test-and-lint`** и **`deploy-simulation`**.

### Работа 1: `test-and-lint` (Непрерывная интеграция)
Эта работа выполняется всегда при срабатывании триггера на виртуальной машине под управлением `ubuntu-latest`. Она состоит из следующих шагов:

1.  **Checkout code:** Клонирование исходного кода репозитория в окружение раннера.
2.  **Setup PHP:** Установка PHP версии `8.2` со всеми необходимыми расширениями (включая SQLite для базы данных в памяти) и настройка быстрого драйвера покрытия тестов `pcov`.
3.  **Copy .env.ci:** Копирование специализированного файла окружения для CI/CD-тестов (`.env.ci`) на место основного файла `.env`.
4.  **Install Dependencies:** Скачивание и установка PHP-зависимостей проекта через Composer в оптимизированном режиме для серверов.
5.  **Generate Application Key:** Генерация уникального ключа шифрования приложения (`APP_KEY`).
6.  **Run PHPStan (Static Analysis):** Запуск статического анализатора кода **Larastan (PHPStan)** на уровне строгости `1`. Любая обнаруженная логическая ошибка приводит к остановке пайплайна.
7.  **Run Laravel Pint (Code Style):** Запуск официального линтера Laravel Pint в тестовом режиме (`--test`) для проверки соответствия кода стандартам PSR-12 / Laravel. Если код оформлен неровно или содержит лишние пробелы, сборка прерывается.
8.  **Run Tests with Coverage Gate:** Запуск автотестов PHPUnit. Настроен защитный барьер (Gate): если процент покрытия кода тестами опускается ниже **50.0%**, или хотя бы один тест падает, пайплайн завершается ошибкой.

---

### Работа 2: `deploy-simulation` (Симуляция доставки)
Эта работа запускается **только в случае успешного прохождения** всех проверок на первом этапе (`needs: test-and-lint`).

#### Особенности и логика деплоя:
*   **Разграничение сред развертывания:** Скрипт автоматически определяет имя текущей ветки Git и выполняет соответствующую симуляцию:
    *   Ветка `develop` (или `dev`) ➔ Копируется файл `.env.dev` и выводится сообщение: `Deploying to [DEV] with .env.dev`.
    *   Ветка `uat` (или `qa`) ➔ Копируется файл `.env.uat` и выводится сообщение: `Deploying to [UAT] with .env.uat`.
    *   Ветки `main` / `master` (Продакшн) ➔ Копируется файл `.env.prod` и выводится сообщение: `Deploying to [PROD] with .env.prod`.
*   **Ручное подтверждение (Manual Approval) для Prod:**
    Для продуктовых веток настроена защита через **GitHub Environments "production"**. Сборка приостанавливается и ждет физического одобрения (кнопки *Review deployment*) от назначенного мейнтейнера (администратора репозитория), прежде чем запустить симуляцию деплоя на прод.

## 3. Дополнительное задание: Уведомление мейнтейнеров

Для информирования лиц, ответственных за стабильность кода (сопровождающих/мейнтейнеров), в самый конец пайплайна добавлен автоматический шаг:
*   **Create Commit Comment on GitHub:** С помощью встроенного безопасного токена `GITHUB_TOKEN` робот заходит в репозиторий и оставляет подробный отчет о статусе сборки (успешно/ошибка) в виде комментария прямо под коммитом, который вызвал сборку. 
*   Каждый мейнтейнер получает мгновенное уведомление на email и внутри GitHub Link о результатах коммита.
